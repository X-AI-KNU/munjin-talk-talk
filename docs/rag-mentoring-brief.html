문진톡톡 RAG 구조 설명

이 문서는 현재 코드 기준으로 RAG가 어디에서 사용되고, 어떤 데이터를 참고하며, LLM 추출 결과와 Hybrid IR 결과가 어떻게 검증되는지 설명한다.

대상: RAG 기술 검토 기준 브랜치: test 저장 원칙: 운영 최소 JSON + 최소 추적 정보

1. 현재 구현에서 RAG가 의미하는 범위

현재 코드에서 RAG는 두 곳에 나뉘어 있다. 첫 번째는 LLM 추출 전에 참고 문맥을 붙이는 intake RAG이고, 두 번째는 추출된 증상 후보를 표준 증상 데이터와 맞추는 Hybrid IR이다.

intake RAG 환자 발화 텍스트를 기준으로 증상 사전과 질환 설명에서 관련 문맥을 검색한다. 검색 결과는 LLM 추출 프롬프트에 약한 참고 자료로만 들어간다.
Hybrid IR LLM이 추출한 증상 후보를 표준 증상명과 매칭한다. BM25, Titan embedding vector, 직접 명칭 일치 신호를 함께 사용한다.
RAG 결과는 환자가 말한 사실이 아니다. source_quote는 반드시 환자 원문에서 나온 문자열이어야 한다. RAG 문서에 있는 표현을 환자 발화처럼 복사하면 검증 실패로 보아야 한다.

3. 한 환자 답변의 처리 순서

환자 발화 STT 텍스트 안전 플래그 intake RAG LLM 추출 Schema 검증 Hybrid IR 운영 JSON 저장
환자 발화가 텍스트로 들어온다. 현재 배포 구조에서는 프론트에서 Web Speech API로 음성을 텍스트화하고, 답변 텍스트를 백엔드에 보낸다.
안전 플래그를 먼저 확인한다. 객혈, 피 섞인 가래처럼 즉시 직원 확인이 필요한 표현은 별도 플래그로 잡는다.
retrieve_intake_rag_context가 참고 문맥을 만든다. 환자 발화와 질문 유형을 입력으로 받아 alias hint와 증상 참고 문서를 만든다.
LLM 추출 프롬프트에 RAG 문맥을 붙인다. 프롬프트는 RAG 문맥을 환자 사실로 쓰지 말라고 명시한다. source_quote는 환자 원문에서만 뽑아야 한다.
Bedrock Nova 계열 모델이 고정 스키마에 맞춰 추출한다. Q1, Q2, Q3, Q4의 난이도와 역할에 따라 모델 라우팅이 달라질 수 있다.
Pydantic 기반 검증이 결과를 확인한다. 필수 키, enum, 타입, 원문 인용 존재 여부를 확인한다. 실패하면 같은 입력으로 재시도하거나 안전한 오류로 반환한다.
증상 후보는 Hybrid IR을 거친다. LLM이 낸 증상 후보가 표준 증상 데이터와 충분히 맞을 때만 원페이퍼의 불편함 카드로 남는다.
운영에 필요한 최소 JSON만 저장한다. 원페이퍼, 답변, 환자 안내문, 의사 리뷰, 동의 기록, 최소 추적 정보가 S3와 DynamoDB에 분리 저장된다.

LLM은 RAG 문서의 질환명이나 증거 문장을 환자 사실로 가져오면 안 된다. 아래처럼 환자 원문에 있는 표현만 source_quote로 남겨야 한다. 

{
  "spans": [
    {
      "type": "symptom_present",
      "name_ko": "목 불편감",
      "source_quote": "목이 칼칼하고"
    },
    {
      "type": "symptom_present",
      "name_ko": "코막힘",
      "source_quote": "코가 막혀요"
    }
  ],
  "clinical_clues": [
    {
      "type": "onset",
      "source_quote": "어제부터",
      "summary": "어제부터 증상 시작"
    }
  ]
}

4-4. Hybrid IR에서 비교되는 문구

LLM 추출 후보 IR query 구성 비교 대상 기대 결과
목 불편감 목이 칼칼하고 목 불편감 증상 문서의 표준명, alias, 질환 백과 근거 문장 목 통증 또는 인후 자극 계열 후보가 올라와야 한다.
코막힘 코가 막혀요 코막힘 코막힘 표준 증상 문서와 관련 evidence label 일치와 vector 유사도가 함께 잡히면 표준 증상 코막힘으로 채택한다.
이 예제의 핵심 검증점은 “코막힘”이 환자 원문과 표준 증상 데이터 양쪽에서 모두 설명 가능해야 한다는 점이다.

5-2. “열”이 원페이퍼 증상으로 들어가면 안 되는 이유

  • 열은 안 나는데는 현재 증상이 아니라 부정 표현이다.
  • RAG가 열 alias를 찾았더라도, 환자 발화의 의미는 “발열 없음”이다.
  • LLM 추출 결과는 symptom_absent 또는 clinical clue로 표현되어야 한다.
  • 원페이퍼의 “오늘 말한 불편함”에는 active symptom만 들어가야 한다.

{
  "spans": [
    {
      "type": "symptom_absent",
      "name_ko": "열",
      "source_quote": "열은 안 나는데"
    },
    {
      "type": "symptom_present",
      "name_ko": "사레 또는 기침",
      "source_quote": "물 마시면 자꾸 사레가 걸려서 캑캑거려요"
    }
  ],
  "clinical_clues": [
    {
      "type": "trigger",
      "source_quote": "물 마시면",
      "summary": "물 섭취 시 증상 유발"
    }
  ]
}

5-4. 이 예제에서 검토해야 할 RAG 설계 지점

“사레”는 일반 호흡기 증상만으로 처리하기 어렵다. 삼킴 곤란, 흡인 위험, 반복되는 폐렴 같은 문맥과 연결된다. 따라서 RAG 문서가 호흡기계에만 좁게 묶이면 매칭 품질이 흔들릴 수 있다.

이 케이스는 RAG 전문가에게 확인할 가치가 있다. “사레”를 표준 증상으로 둘지, 임상 단서로 둘지, 또는 흡인 위험 관련 별도 clinical clue로 둘지 결정해야 한다.

6. Hybrid IR 점수 산출 방식

UI에는 숫자 점수를 표시하지 않는다. 내부에서는 후보 정렬과 채택 여부를 위해 점수를 계산한다. 이 값은 진단 확률이나 의학적 확신도가 아니다.

6-1. 비교 문구 생성

query_text = normalize_text(
  source_quote + " " + normalized_answer + " " + span_name
)

환자 원문 인용, 표준화 답변, LLM이 붙인 증상명을 함께 사용한다. 한쪽 표현만 사용할 때보다 방언, 짧은 발화, 비문에 덜 흔들리게 하기 위한 구성이다.

6-2. 후보 문서 생성

검색 문서는 diseases_cleaned.json과 symptom_index.json에서 생성된다. 별도의 LLM 가공 데이터셋을 전제로 하지 않는다.

데이터 사용 방식
symptom_index.json 표준 증상명, 증상 ID, alias 후보의 기준으로 사용한다.
diseases_cleaned.json 질환 백과의 증상 설명 문장을 evidence로 사용한다.

6-3. 계산 신호

신호 의미 역할
BM25 단어 기반 검색 점수 환자 표현과 문서 문구가 직접적으로 겹치는지 본다.
Titan vector 의미 기반 유사도 단어가 정확히 같지 않아도 의미가 가까운 후보를 찾는다.
label score 표준명 또는 alias 직접 일치 “코막힘”처럼 명확한 직접 표현이 있을 때 후보를 보강한다.

6-4. 내부 rank score

rank_score =
  0.35 * bm25_norm
  + 0.65 * vector_norm
  + 0.25 * label_score
  + preferred_alias_bonus
  + agreement_bonus
  - bm25_only_penalty

rank_score는 후보 정렬용이다. 화면에 보여주는 의료적 점수가 아니다. 현재 UI에서는 숫자를 숨기고 “매칭됨” 또는 표준 증상명 중심으로 보여주는 방향이 맞다.

6-5. 채택 gate

accepted =
  vector_score >= 0.12
  and (
    bm25_score >= 0.04
    or label_score >= 0.55
  )
BM25만 높고 vector가 거의 맞지 않는 후보는 채택하면 안 된다. 반대로 vector만 높고 원문 또는 표준명 근거가 약한 후보도 검토 대상이다.

7. 사투리 RAG 추가 설계

현재 사투리 표준화는 LLM의 일반 언어 이해에 많이 의존한다. 강원 방언 문맥을 안정적으로 처리하려면 별도 RAG 노드가 필요하다.

7-1. 추가될 위치

환자 발화 안전 플래그 사투리 RAG intake RAG LLM 추출 검증

7-2. 방언 RAG 데이터 예시

{
  "dialect_term": "마카",
  "standard_candidates": ["모두", "전부"],
  "region": "강원",
  "examples": [
    {
      "dialect_sentence": "열은 마카 싹 내렸소.",
      "standard_sentence": "열은 전부 싹 내렸습니다."
    }
  ],
  "notes": "문맥에 따라 강조 부사로만 처리한다. 증상명으로 추출하지 않는다."
}

7-3. 허용 규칙

  • 방언 RAG는 표준어 변환 후보만 제공한다.
  • 방언 RAG는 증상, 질환, 처방, 검사를 새로 만들지 않는다.
  • 방언 예문은 source_quote로 사용할 수 없다.
  • 최종 source quote는 환자가 실제 말한 원문에서만 가져온다.

7-4. 남겨야 할 최소 추적 정보

필드 필요 이유
dialect_matches 어떤 방언 표현이 표준화에 참고되었는지 확인한다.
standardization_rationale 표준화가 환자 의미를 바꾸지 않았는지 검토한다.
source_quote_map 원문 표현과 표준화 표현의 연결을 확인한다.

8. 저장해야 할 RAG 관련 최소 정보

운영 저장은 서비스 실행과 사후 검토에 필요한 최소 정보만 남긴다. 디버깅용 전체 프롬프트, 불필요한 boolean 플래그, 화면에서 쓰지 않는 중간값은 운영 저장 대상이 아니다.

정보 저장 여부 이유
환자 원문 답변 저장 의료진 확인과 source quote 검증에 필요하다. 가명 처리와 보존 기간 제한이 필요하다.
표준화 답변 저장 원페이퍼와 안내문 생성에 필요하다.
추출 span과 source quote 저장 LLM 결과가 원문에 근거했는지 확인한다.
선택된 표준 증상 ID와 표준명 저장 원페이퍼 표시와 추후 통계에 필요하다.
채택된 evidence 요약 최소 저장 왜 해당 표준 증상으로 매칭됐는지 설명하기 위한 최소 근거다.
전체 RAG 후보 목록 운영 저장 제외 화면과 운영에는 불필요하다. 필요하면 개발 환경 로그에서만 본다.
전체 프롬프트와 전체 LLM 응답 원문 운영 저장 제외 개인정보와 민감정보 노출 위험이 크다.

9. RAG 검토 질문

  • LLM 추출 전에 넣는 weak RAG 문맥이 현재 수준으로 충분한가.
  • 방언 RAG를 별도 노드로 분리할 때, 표준화 후보와 의료 증상 후보를 어떻게 분리할 것인가.
  • “사레”, “숨 넘어갈 것 같음”, “기운 없음” 같은 표현을 증상으로 둘지 clinical clue로 둘지 기준이 필요한가.
  • 질환 백과 문장의 evidence를 증상 매칭에 사용할 때 오매칭 위험을 어떻게 줄일 것인가.
  • RAG 후보를 얼마나 저장해야 설명 가능성과 개인정보 최소 저장 원칙이 동시에 충족되는가.
