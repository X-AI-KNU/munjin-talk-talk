문진톡톡 MVP 기술 안내서

프론트, 백엔드, AI 처리, JSON 저장 구조를 한 번에 보는 문서

이 문서는 현재 배포된 문진톡톡 MVP가 실제로 어떻게 동작하는지 설명합니다. 환자 음성 문진이 Amazon Transcribe, Bedrock Nova, Titan embedding, Lambda, DynamoDB, S3 artifact bucket을 거쳐 의사용 원페이퍼와 환자 안내문으로 바뀌는 전체 흐름을 기준으로 정리했습니다.

Frontend React/Vite 화면, 녹음, API 호출, 원페이퍼와 안내문 표시
Serverless Backend API Gateway, Lambda, DynamoDB 상태 관리, S3 artifact 저장
AWS AI Transcribe STT, Nova 의미 분해, Titan vector 검색
Output 의사용 원페이퍼, 의료진 확인 항목, 환자용 안내문

핵심 문장: DynamoDB에는 세션 상태와 S3 key만 남기고, 문진 답변·원페이퍼·안내문·trace는 가명처리 후 S3 artifact로 분리 저장합니다.

요약 실행 흐름 폴더 구조 프론트엔드 백엔드 AI 역할 증상 매칭 JSON 스키마 배포 구조 확인 방법

Current MVP

1. 현재 구현 기준 요약

지금 코드는 단순 데모 UI가 아니라, AWS Transcribe와 Bedrock을 실제 호출하는 서버리스 MVP 구조입니다. 제출 환경은 직원/의료진 접근 코드, S3 artifact 분리 저장, DynamoDB TTL, WAF, CloudTrail, GuardDuty, Security Hub, Macie 기준으로 정리되어 있습니다.

접수처

직원이 환자 이름, 생년월일, 성별, 진료과, 담당 의사, 연락처를 입력하고 문진 세션을 생성합니다.

환자 태블릿

환자가 초진/재진 흐름에 따라 Q1~Q4에 음성으로 답하고, 실시간 인식 텍스트를 보며 진행합니다.

의사 원페이퍼

증상, 원문 quote, 문진 맥락, 환자 질문, 의료진 확인 항목을 한 화면에서 확인합니다.

환자 안내문

의사가 환자 질문에 답하면 Nova Lite가 쉬운 표현으로 정리하고, 의사 강조사항은 원문 그대로 별도 표시합니다.

데이터 저장

DynamoDB에는 대기열과 상태, S3 artifact key만 저장됩니다. 원페이퍼와 안내문은 가명처리 후 S3에 저장하며, 환자 음성 파일은 저장하지 않습니다.

안전 플래그

객혈 등 우선 확인 표현은 Lambda rule로 감지하고, 환자 화면에서는 직원 도움 흐름으로 전환됩니다.

중요한 정정: 현재 증상 매칭은 단순 단어 포함 점수가 아닙니다. 배포 환경에 비공개로 배치한 diseases_cleaned.json과 symptom_index.json에서 rule-base로 만든 증상 문서를 대상으로 BM25 lexical 검색, Titan embedding vector 검색, LLM이 추출한 slot_ref/name 힌트를 함께 반영합니다. 생성형 LLM이 IR 안에서 새 증상을 만들어내는 구조는 아닙니다.

Runtime Flow

2. 환자 1명이 진행할 때의 실제 흐름

사용자가 화면에서 보는 흐름과 백엔드 내부 처리 순서를 같은 순서로 놓으면 아래와 같습니다.

1. 직원 접수 POST /sessions
DynamoDB에 기본 session 생성
2. 태블릿 진입 /patient/{id}
초진/재진 문항 시작
3. 실시간 STT URL /transcribe-stream-url
Transcribe Streaming 서명 URL 발급
4. 실시간 STT 브라우저 WebSocket
음성 저장 없이 text만 수신
5. 서버 파이프라인 /process-answer
LangGraph 문항 처리 시작
6. RAG 참고 문맥 원천 JSON + 제한 alias
LLM 표준화 참고용
7. LLM + 검증 LangChain Runnable
Pydantic/source_quote retry
8. 증상 매칭 Lambda 내부 graph
BM25 + Titan hybrid IR
9. 저장 검증 Lambda 내부 graph
S3 artifact 저장 + DDB 상태 갱신
10. 원페이퍼 /onepager/{id}
Nova Pro가 의사 task 생성
LangChain/LangGraph 적용 방향: LangGraph는 실제 StateGraph로 문항 처리 노드를 연결합니다. LangChain은 ChatPromptTemplate -> Bedrock RunnableLambda -> JsonOutputParser 체인으로 Nova 호출을 감싸며, RAG 참고 문맥 검색과 Pydantic/source_quote validator는 각각 별도 graph node로 trace됩니다. 검증 실패 시 graph가 schema_quote_validation -> semantic_extraction retry edge를 탑니다.

Q별 처리 차이

  • Q1: 주 증상/재진 경과 문항이라 Nova Pro를 사용합니다.
  • Q2~Q4: Nova Lite를 사용하지만, 표준화와 의미 분해는 모두 수행합니다.
  • Q1, 재진 경과, 새 증상 문항에서 symptom-like span이 있으면 LangGraph 내부 hybrid_ir_match 노드가 실행됩니다.
  • Q4의 환자 질문은 structured.questions와 원페이퍼 agenda로 이어집니다.

진료 후 흐름

  • 의사가 원페이퍼에서 환자 질문별 답변을 입력합니다.
  • POST /doctor-response가 답변과 강조사항을 저장합니다.
  • Nova Lite가 환자용 안내문 문장을 쉽게 다시 씁니다.
  • 의사 강조사항은 LLM이 바꾸지 않고 안내문에 그대로 표시합니다.

AI Orchestration

7. LLM과 AI 호출 역할

역할별로 모델을 완전히 다른 서비스처럼 분리한 것은 아니지만, 문항 난이도와 업무 성격에 따라 Nova Pro, Nova Lite, Titan embedding을 나누어 호출합니다.

단계 모델/서비스 하는 일 JSON 영향
음성 인식 Amazon Transcribe 브라우저 WebSocket으로 환자 음성을 보내고 텍스트만 수신 S3 answers.redacted.json
RAG 참고 문맥 Local reference retriever 원천 JSON과 제한 alias bridge에서 표준화 참고 문맥 검색. 환자 사실로 직접 채택하지 않음 S3 llm_trace.redacted.json의 최소 trace
LLM 호출 체인 LangChain Core Runnable ChatPromptTemplate -> Bedrock RunnableLambda -> JsonOutputParser로 Nova 호출과 parser를 trace S3 llm_trace.redacted.json의 model/parser 요약
Q1 의미 분해 Bedrock Nova Pro 주 증상 또는 재진 경과를 증상 span, 맥락, 표준 문장으로 구조화 spans, structured.standardized_text, clinical_clues
Q2~Q4 의미 분해 Bedrock Nova Lite 발병 시점, 복약, 환자 질문을 표준화하고 구조화 clinical_clues, structured.questions
스키마/quote 검증 Pydantic validator 필드, enum, extra field, 원문 substring을 검증하고 실패 시 LangGraph retry edge 실행 validator_passed, validation_errors
증상 IR BM25 + Titan embedding LLM이 추출한 증상 span을 실제 표준 증상 문서와 매칭 운영 matched_slots, 감사용 llm_trace.redacted.json
의사용 task 생성 Bedrock Nova Pro 증상 JSON, 환자 질문 JSON, safety flag, 원문을 보고 의사가 확인할 다음 행동 생성 onepager.review_items, doctor_brief, transfer_text
환자 안내문 생성 Bedrock Nova Lite 의사 답변을 어르신 환자에게 쉬운 안내문으로 재작성 patient_guide.items
안전 감지 Lambda rule 객혈, 심한 호흡곤란 등 우선 확인 표현 감지 safety_flag, safety_flags, risk

1명 기준 대략 호출 수

  • Transcribe Streaming: Q1~Q4 질문마다 WebSocket 세션 1회, 총 4회
  • Nova Pro: Q1 의미 분해 1회, 원페이퍼 review 1회
  • Nova Lite: Q2~Q4 의미 분해 3회, 안내문 생성 1회
  • Titan embedding: 매칭할 증상 span 수만큼 보통 2~5회

LLM 실패 정책

  • LLM extraction 실패 시 rule-base 결과로 조용히 대체하지 않습니다.
  • 원페이퍼 의료진 확인 항목은 Nova Pro review가 schema와 근거 검증을 통과한 경우에만 저장합니다.
  • 환자 안내문은 Nova Lite 결과가 너무 일반적이거나 원문 복사면 validator 실패로 표시합니다.

Symptom Retrieval

8. 증상 매칭과 내부 근거 계산

원페이퍼의 증상 매칭은 LLM이 임의로 확신도를 찍는 값이 아니라, LangGraph 내부 hybrid_ir_match 노드에서 BM25, Titan vector, 표준명/alias bridge를 함께 계산한 결과입니다. 의료진 UI에는 숫자 점수 대신 “매칭됨” 또는 “우선 확인” 같은 상태로 표시합니다.

8-1. 검색 문서 생성

diseases_cleaned.json
  - 비공개 배치 파일. 질환 백과 기반 질환/증상 설명

symptom_index.json
  - 비공개 배치 파일. 표준 증상명과 질환 문서 연결 인덱스

domain_packs/respiratory.json
  - 호흡기계 MVP에서 허용하는 symptom slot
  - 제한 alias bridge, safety flag, 기본 질문 문구

Lambda cold-start
  - 원천 JSON과 도메인팩을 읽어 증상별 검색 문서 생성
  - 별도 LLM 가공 데이터셋은 사용하지 않음

symptom_embeddings_...json
  - 비공개 배치 파일. 같은 증상 문서를 Titan으로 미리 임베딩한 벡터 인덱스

확장성 의미

호흡기계만 손으로 조합한 폐쇄 구조가 아닙니다. 같은 구조의 비공개 원천 데이터와 domain_packs/*.json, question_sets/*.json를 준비하면 다른 진료계 질환도 같은 방식으로 확장할 수 있습니다.

8-2. 내부 계산 요소

요소 의미 예시
bm25_score 환자 표현과 증상 문서가 글자/토큰 수준에서 얼마나 맞는지 보는 lexical 검색 “기침”, “콜록”, “목이 칼칼” 같은 표현 매칭
vector_score Titan embedding으로 의미 벡터 거리를 비교한 점수 단어가 완전히 같지 않아도 의미가 가까운 증상을 찾음
label_score LLM이 추출한 slot_ref, name이 표준명/alias와 얼마나 맞는지 보는 힌트 slot_ref=cough이면 기침 후보를 우선 고려
rank_score 후보 정렬용 종합 점수 BM25 0.35, vector 0.65, label hint를 함께 반영
LLM이 “이건 기침이다”라고 말하면 끝나는 구조가 아닙니다. LLM의 slot_ref는 검색 질의의 힌트이고, 최종 표준 증상명은 BM25 + Titan 검색 후보 중에서 선택됩니다. 다만 preferred_alias가 강하게 맞으면 후보 정렬이 크게 영향을 받을 수 있으므로, 이상하면 S3 answers.redacted.json의 운영 매칭 결과와 llm_trace.redacted.json의 확정 매칭 근거 요약을 함께 확인합니다.

      "standardized_text": "어제부터 목 불편감과 코막힘이 있음",

중요 검증 규칙

  • source_quote와 original_quote는 환자 발화의 실제 부분 문자열이어야 합니다.
  • Pydantic fixed schema가 타입, enum, 필수 필드, 추가 필드 금지를 기계적으로 검증합니다.
  • LLM이 없는 말을 만들면 quote repair/validation에서 버려집니다.
  • LLM extraction JSON에는 score, confidence, probability 같은 임의 수치를 저장하지 않습니다.
  • LLM이 임의 수치 필드를 넣으면 extra_forbidden 오류로 retry loop에 들어갑니다.
  • 운영 artifact와 의료진 UI에는 숫자 매칭 점수를 저장하거나 표시하지 않습니다.
  • BM25, Titan vector, 표준명 유사도는 확정 매칭의 최소 설명 trace에만 요약됩니다.
  • standardized_text는 Q1뿐 아니라 Q2~Q4에도 생성됩니다.
  • DynamoDB에는 이 문항 원문과 spans가 직접 저장되지 않고 S3 artifact key만 남습니다.

Verification

11. 이상할 때 확인하는 순서

UI가 이상해 보이면 프론트 문제가 아니라 session JSON 생성 단계 문제일 수 있습니다. 아래 순서로 보면 원인을 빠르게 좁힐 수 있습니다.

증상 먼저 볼 곳 확인할 JSON/서비스
환자 목록이 안 보임 DynamoDB MunjinSessions에 session item이 있는지 확인
음성이 텍스트로 안 바뀜 Transcribe Streaming, 브라우저 콘솔 /transcribe-stream-url 200 여부, 마이크 권한, WebSocket 오류, Lambda IAM 권한
원문 quote가 이상함 S3 answers.redacted.json, llm_trace.redacted.json source_quote, original_quote, validator event의 source_quote_grounding
증상명이 이상함 S3 answers.redacted.json, llm_trace.redacted.json 운영 matched_slots와 trace의 accepted_matches 근거 요약
환자 질문이 합쳐짐 structured.questions Q4에서 질문이 배열로 분리됐는지 확인
의료진 확인 항목이 부자연스러움 onepager.review_items review_item_generation.method가 bedrock_nova_pro인지 확인
안내문이 원문 그대로임 S3 patient_guide.redacted.json generation_method, S3 doctor_review.redacted.json, additional_notes

프론트 확인

브라우저 개발자도구 Network에서 API 요청/응답과 status code를 봅니다.

백엔드 확인

CloudWatch Logs의 Lambda log group에서 예외와 Bedrock/Transcribe 오류를 확인합니다.

데이터 확인

DynamoDB item은 상태와 artifact key만 보고, 실제 답변/원페이퍼/안내문은 S3 redacted JSON을 봅니다.

One Sentence

마지막으로 한 문장 정리

문진톡톡은 환자 발화 텍스트를 가명처리 artifact JSON으로 구조화하고, 그 JSON을 기반으로 의사에게는 확인할 task를, 환자에게는 쉬운 안내문을 제공하는 AWS 서버리스 문진 MVP입니다.
